# 可信人格记忆Agent 平台说明

本文档面向第一次使用可信人格记忆Agent的开发者或评测人员，说明当前准备事项、启动方式、验证方式和常见结果。当前后端覆盖 Milestone 0-8 的健康检查、认证/demo API、人物、资料、任务、Memory Audit v2 后端 API、档案、单资料严格六模块 JSON 记忆抽取、上传解析后的自动档案摘要和唯一可信度、对话、声音、3D、stories/export/delete/provider settings API，本地资料文件清理、长期/短期记忆 Markdown 和可选 DashScope/MiniMax/OpenAI-Next provider 接入。当前前端以 `origin/yawen` 的星空交互为准：全局顶部导航固定为 `首页 / 产品介绍 / 创建档案 / 我的星空`，人物详情页下方资料、记忆和互动功能卡片作为人物内功能入口，各人物内功能页不再展示人物工作台块、分组按钮或返回人物工作台按钮；展示星空首页、Dashboard、创建星星、人物详情、资料解析与审核、任务、记忆档案馆、对话、遗憾对话室、心愿延续系统、声音和 3D 形象页。对话页右侧提供数字人舞台，形象页上传 GLB 后会替换默认星空占位；uploads 页展示四宫格资料上传、结构化记忆 Markdown、唯一“记忆可信度”、6 个维度审核卡片、记忆星标和“完成审核，点亮星星”按钮，不再展示手动档案摘要卡片；memories 页只保留回忆讲述和语义搜索，进入页面会幂等生成最多三段锚定真实已审核记忆的默认回忆故事，用户输入关键词后可追加新故事，story/search 会读取长期/短期记忆辅助检索，故事正文会清洗模型思考块，搜索仍只返回可追溯 MemoryCard；`/personas/{id}/profile` 作为兼容深链重定向到 uploads，不再展示独立人格档案页面。不展示登录/注册、登录态差异、数据设置页、模型设置页或独立回忆讲述页。无 token 时前端会透明调用现有 `POST /api/auth/demo` 获取 demo token；已有 token 静默复用。真实解析质量需要本地配置 `DASHSCOPE_API_KEY` 后测试；真实文本 Chat/Story LLM、Persona Engine 显式画像分析、结构化记忆文档生成和记忆上下文压缩已可使用本地 MiniMax/OpenAI-compatible 配置，MiniMax 文本失败时可使用本地 OpenAI-Next text-only fallback，长期人格质量、真实 TTS/音色质量、真实 3D provider 质量、真实 MinIO/S3 对象删除和生产级 secret manager 仍待后续里程碑接入。

补充：embedding 运行入口已停用并移除；历史数据库 embedding 列和 Alembic `0004` 保留以兼容旧库。当前普通对话上下文链路为“LLM/规则抽取结构化记忆 -> 已确认/已修正记忆生成长期 Markdown -> kind 维度短期对话 Markdown -> Chat 直接读取 Markdown”。Markdown 过长时通过 `memory_context_compression` 压缩，失败时用确定性截断兜底，不阻断聊天。遗憾对话室使用 `kind=regrets` 专用 conversation 和“有没有什么以前没说的话，今天想慢慢告诉我？”system prompt，保留长期已确认记忆但只读取 regrets 同类短期 Markdown，不带入普通聊天消息。心愿延续页使用 `context_kind=wishes` 专用 conversation 和心愿 system prompt，只读取当前心愿对话历史、人物基础信息和语气资料，不带入普通聊天、遗憾对话等其他 conversation 的短期 Markdown。

当前补充：记忆审核和点亮星星已放入 `/personas/{id}/uploads`；回忆讲述和语义搜索放入 `/personas/{id}/memories`；对话页 `/personas/{id}/chat` 只保留文字、手势、语音对话和侧边数字人；`/personas/{id}/regrets` 与 `/personas/{id}/wishes` 是独立前端引导页，会分别创建/复用 `kind=regrets` 与 `context_kind=wishes` 专用对话并使用专门 system prompt，不代表已实现独立遗憾记录模型、心愿数据模型、提醒策略或长期 P1 心愿系统。

## 适用对象与使用前准备

### 适用对象

- 开发者：验证本地基础工程、后端测试、前端检查、Compose 配置、Milestone 1-8 后端/API 基础闭环，以及当前 yawen 星空前端的首页、Dashboard、创建星星、人物详情、资料上传、任务、记忆档案馆、chat、regrets、wishes、voice 和 avatar 页面。Milestone 8 stories/export/delete/provider settings 后端 API 保留，但当前不暴露独立前端设置页、stories 页或 profile 页 UI。
- 评测人员：确认当前是否具备前后端基础连通、健康检查、注册/登录 API、人物 CRUD、人物列表、人物创建表单校验、人物记忆空间页、资料上传页、任务状态页、记忆档案馆、自动档案摘要/唯一可信度和文本对话页。
- 产品体验者：当前可以从星空首页进入体验，无需主动登录或注册；系统会在需要鉴权时自动创建 demo session。全局顶部导航只负责进入首页、产品介绍、创建档案和我的星空；进入某个人物后，通过人物详情页下方资料、记忆和互动功能卡片进入具体功能页。前端可创建星星、上传珍贵回忆、查看资料任务、在资料解析与审核页点亮/审计记忆、在记忆档案馆生成几段回忆讲述或做语义搜索、进入文字/手势/语音对话、使用遗憾对话室、心愿延续引导、声音设置和 3D 形象设置。上传资料会生成 deterministic mock 记忆卡片供确认、修改、删除或星标，uploads 页以 6 个维度卡片组织可见记忆；上传解析链路会自动生成结构化记忆 Markdown、`profile_summary` 和唯一 trust 值，后续记忆审核只刷新 profile 维度/长期 Markdown，不另算第二套可信度，也不覆盖自动摘要。人物详情页顶部简介读取最近一次上传解析自动生成的档案摘要，没有摘要时显示中性空态。文本对话会使用 confirmed/corrected 记忆并展示引用依据，星标记忆优先进入上下文；配置 MiniMax 后会通过真实 `chat_llm` 生成回复。对话页侧边数字人舞台会优先展示已选择/生成的 GLB 数字人，未设置时展示星空占位。视频手势模式会在浏览器本地开启摄像头识别停留、挥手、张开手掌和握拳，并驱动 GLB 自带动画或基础动作反馈，不上传或保存摄像头画面；遗憾对话室使用 `kind=regrets` 专用对话和“有没有什么以前没说的话，今天想慢慢告诉我？”主题 prompt，不带入普通聊天短期上下文；心愿延续引导使用专用 `context_kind=wishes` 对话和“你现在有什么想完成的心愿，或者想替我继续做的一件事吗？”主题 prompt，不带入其他聊天上下文，但不代表已实现独立后端遗憾记录模型、心愿数据模型、提醒策略或长期 P1 心愿系统。真实文本 LLM、TTS/音色和 3D provider 质量仍需用更大样本单独验收。

### 使用前准备

当前需要具备：

- 能打开本仓库。
- Python 3.12 或兼容环境，可安装 `backend/requirements.txt`。
- Node.js 和 npm；在 PowerShell 中运行 npm 命令时使用 `npm.cmd`。
- Docker Desktop 或兼容 Docker Compose，用于验证 Compose 配置和本地服务拓扑。
- 如部署到阿里云 ECS，需要 Docker Compose 可用、ECS 安全组放行 TCP `3000` 和 `8000`，并准备实际公网 IPv4 用来替换文档中的 `<ECS公网IP>` 占位符。

不需要提交真实密钥。Compose 和后端配置只使用变量名、空白 provider 配置和开发默认值；真实 provider key 应只放在本地未提交环境中。默认 `DEFAULT_LLM_PROVIDER=mock` 不会调用第三方；如需真实解析，可在 `.env/runtime.env` 中配置 `DEFAULT_LLM_PROVIDER=dashscope`、`DASHSCOPE_API_KEY`、`DASHSCOPE_WORKSPACE_ID`、`DASHSCOPE_COMPAT_BASE_URL`、`QWEN_TEXT_MODEL`、`QWEN_VISION_MODEL`、`QWEN_OCR_MODEL`、`QWEN_ASR_MODEL` 和可选的 `DASHSCOPE_REQUEST_TIMEOUT_SECONDS`，默认超时为 180 秒。如需真实文本 Chat/Story LLM，使用本地 MiniMax 配置：`MINIMAX_API_KEY`、`MINIMAX_BASE_URL=https://api.minimaxi.com/v1`、`OPENAI_BASE_URL=https://api.minimaxi.com/v1`、`OPENAI_API_KEY` 和 `OPENAI_MODEL`；如需 MiniMax 文本失败兜底，再配置 `OPENAI_NEXT_API_KEY`、`OPENAI_NEXT_BASE_URL=https://api.openai-next.com/v1`、`OPENAI_NEXT_MODEL=gpt-5` 和可选 `OPENAI_NEXT_REQUEST_TIMEOUT_SECONDS=60`。OpenAI-Next 只用于文本生成相关 capability，不用于多模态解析、TTS 或音色克隆。响应和文档只显示配置状态，不输出原始 key。当前前端不暴露模型设置页；如需查看或更新 provider configured/missing 状态，可通过后端 `GET/PUT /api/settings/providers` 或本地 `.env/runtime.env` 处理，响应只返回遮蔽后的配置状态，不返回原始 API key。

如需执行真实多模态公网样本烟测，先启动后端，再运行：

```powershell
python backend/scripts/real_multimodal_smoke.py --sample-mode public --backend-url http://localhost:8000
```
该命令会下载公网公版样本、派生 PDF/DOCX/DOC、本地裁剪视频，并真实调用 DashScope；结果写入 `.smoke/real-multimodal/results/`，不会提交或输出原始密钥。当前本地最近一次完整烟测已在 2026-07-04 通过，结果为 `.smoke/real-multimodal/results/20260704-170719.json`：text、image、audio、video、pdf、docx、doc 均生成 DashScope third_party parse job、ParsedChunk 和 source-backed MemoryCard，`errors=[]`。如果复跑时再次出现 `403 AccessDenied.Unpurchased` / `Access to model denied. Please make sure you are eligible for using the model.`，先在阿里云百炼控制台确认账号已开通百炼服务、完成实名认证、业务空间/API Key 与当前 region 匹配，并确认 `QWEN_TEXT_MODEL`、`QWEN_VISION_MODEL`、`QWEN_OCR_MODEL`、`QWEN_ASR_MODEL` 对该 key 可调用；开通或替换为有权限的模型后重启 backend 再复跑。

### 启动与访问

安装前端依赖：

```powershell
npm.cmd --prefix frontend install
```

如用户明确要求完整基线验证，可运行：

```powershell
& "C:\Program Files\Git\bin\bash.exe" ./docs/init.sh
```

启动本地服务拓扑：

```powershell
docker compose up --build
```

后端容器启动时会先执行 `alembic upgrade head`，再启动 Uvicorn 服务；因此 Compose 中的注册/登录 API 会使用已迁移的 PostgreSQL 表结构。

默认访问地址：

- 前端：http://localhost:3000
- 后端健康检查：http://localhost:8000/health
- MinIO Console：http://127.0.0.1:9001

ECS 直连端口部署时，在 ECS 仓库根目录创建未提交的 `.env/runtime.env`，并把 `<ECS公网IP>` 替换成阿里云 ECS 实际公网 IPv4，不要保留尖括号：

```env
FRONTEND_URL=http://<ECS公网IP>:3000
BACKEND_URL=http://<ECS公网IP>:8000
NEXT_PUBLIC_API_BASE_URL=http://<ECS公网IP>:8000
JWT_SECRET=<生成强随机字符串>
```

然后运行：

```bash
docker compose --env-file .env/runtime.env up --build -d
```

远程访问地址：

- 网页：`http://<ECS公网IP>:3000`
- 后端健康检查：`http://<ECS公网IP>:8000/health`

`NEXT_PUBLIC_API_BASE_URL` 会在 Next.js 构建期写入客户端包；公网 IP 或后端端口变化后必须重新带 `--build` 启动。ECS 安全组不要对公网放行 `15432`、`6379`、`9000`、`9001`，当前 Compose 默认把 PostgreSQL、Redis 和 MinIO 宿主端口绑定到 `127.0.0.1`。

## 用户全流程操作步骤

以下只描述当前可验证流程。完整人格记忆产品流程需在后续里程碑实现后补齐。

### 1. 进入系统

启动 Compose 后，本地调试打开 http://localhost:3000 查看星空首页，ECS 远程访问打开 `http://<ECS公网IP>:3000`；本地访问 http://localhost:8000/health 或远程访问 `http://<ECS公网IP>:8000/health`，返回 `{"status":"ok"}` 表示后端健康检查可用。当前顶部导航只有 `首页`、`产品介绍`、`创建档案` 和 `我的星空`，其中 `产品介绍` 进入 `/product-intro`，不会在全局导航中展示记忆审核或故事类功能入口。当前前端不提供 `/login` 或 `/register` 页面；无 token 时，创建和核心工作流会自动调用 `POST /api/auth/demo` 创建本地演示用户，无需邮箱或密码。

### 2. 配置或确认人格记忆范围

当前已实现人物创建基础配置。进入 `/dashboard` 可查看当前 token 对应的人物列表；进入 `/personas/new` 可创建已故亲友、在世亲友或虚拟角色，并填写姓名、年龄/享年、简介、TA 对用户的角色和 TA 对用户的称呼。创建页内部展示 `资料卡片 -> 上传回忆 -> 保存并进入审核` 阶段流程，这些阶段不进入全局顶部导航。主要语言固定为中文；说话风格、情绪陪伴方式和禁用表达由系统默认值写入。缺少必填字段或年龄/享年超出 1..150 时，前端会阻止提交；后端也会拒绝缺少或无效的 age，并拒绝非 `zh-CN` 的语言值。创建成功后，创建页会把已选择的照片、视频、声音和文字文件通过现有资料上传 API 补传，并进入 `/personas/{id}/uploads`；若上传失败，已创建人物会保留，用户可进入资料上传页继续补传。

### 3. 输入或导入记忆材料

当前已实现资料记录、默认 mock 解析和可选 DashScope 真实解析。登录或进入演示后，可从人物详情页下方资料卡片进入 `/personas/{id}/uploads`，上传文本、图片、音频、视频文件，也可填写手动资料。提交后接口会先创建 `SourceMaterial` 和 pending parse job 并快速返回，页面在按钮下方显示真实 job 状态进度并轮询 materials/memories/profile；后台解析完成后才出现新的审核卡片和更新后的结构化文档。上传文件会写入仓库本地 `storage/materials/...` 兜底路径；默认通过 deterministic mock Provider Gateway 创建 `ParsedChunk`，再把单个资料的识别文本送入严格六模块 JSON `memory_extraction`，后端校验 `structured_memory_json` 后派生带来源的 `MemoryCard`。配置 `DEFAULT_LLM_PROVIDER=dashscope` 与 `DASHSCOPE_API_KEY` 后，文本/PDF/DOC/DOCX 抽取、图片 OCR/理解、音频 ASR、视频音轨/关键帧解析和严格模块记忆抽取会走 DashScope adapter。

### 4. 查看记忆提取结果

当前已实现记忆提取结果、资料审核和回忆讲述展示。进入 `/personas/{id}/uploads` 可查看四宫格资料上传区、由当前 active `MemoryCard` 渲染的结构化记忆 Markdown、唯一“记忆可信度”、6 个维度审核卡片、记忆星标和“完成审核，点亮星星”按钮；系统从资料中整理出的真实后端记忆来自严格模块 JSON，按基础信息、人物关系、兴趣偏好、生活习惯、表达习惯和共同经历进入这些维度卡，确认/编辑/删除/星标由前端调用现有记忆 API。结构化文档和下方审核卡片使用同一批 MemoryCard，因此不会从 `persona_card` 或 parsed chunks 直接补出不可审核的条目。资料卡片只展示真正资料解析失败原因；结构化文档 provider 格式诊断只保存在 job metadata，不在用户界面展示。进入 `/personas/{id}/memories` 可查看星空主题“记忆档案馆”，该页只保留“让TA讲几段回忆”和语义搜索；页面会先调用 seed 接口幂等补足最多三段默认回忆故事，每段锚定一条已确认或已修正记忆，用户输入主题后点击按钮会追加关键词对应的新故事；回忆讲述读取长期/短期记忆并展示中文故事正文、来源标题和来源位置，语义搜索在长期/短期记忆中查找可追溯来源记忆，结果内联展示标题、分类、来源摘录和位置，不展示短期对话片段。进入 `/personas/{id}/jobs` 可查看资料对应的 AI Job 状态、任务类型、重试次数、provider 和错误信息；默认 mock 与真实 provider 解析都通过后台 parse job 执行，真实质量需用有效 key 和真实材料单独验收。

### 5. 复核、修正或删除记忆

当前已实现记忆复核、修正、删除、星标和审计追踪。进入 `/personas/{id}/uploads` 后，可以确认、编辑内容、删除或星标记忆；编辑会保存为用户修正后的内容，星标会写入 `MemoryCard.is_important`。Memory Audit v2 会在同一事务中记录创建、编辑、确认、拒绝、停用、删除、搜索、对话引用、对话纠错、故事引用、profile 编辑/重生成和可信度变化事件；相关后端 API 保留。记忆审核动作会刷新人格档案维度和长期 Markdown，但不覆盖上传解析链路生成的自动摘要或 trust；confirmed/corrected 记忆会写入该人物长期记忆 Markdown，星标记忆优先进入长期 Markdown、故事和对话上下文，rejected/disabled/deleted 记忆不会进入长期 Markdown 或对话引用。

### 6. 查看和编辑档案摘要

当前已实现 deterministic local 人格档案数据能力和 Persona Engine 显式画像分析，但不再提供独立 `/personas/{id}/profile` 前端编辑页；访问该深链会进入 `/personas/{id}/uploads`。uploads 不再提供“档案摘要”手动编辑卡片；每次资料解析后，后端先用当前 active MemoryCard 确定性生成 `structured_memory_document_json` 和 `structured_memory_md`，再由 `memory_document_generation` 自动输出 `profile_summary`、trust、理由和建议。MiniMax 返回代码块、`<think>` 或前后说明包裹的 JSON 时会先清洗再校验，仍不可用时保留已生成的记忆卡片和同源结构化文档，并仅保留内部诊断。显式 profile 重生成仍可调用 `persona_profile_analysis` capability，输入 persona card、已解析 chunk、active memory cards 和 source metadata，保存完整 `persona_engine_json` 与生成时间，但不覆盖上传解析链路生成的自动摘要或 trust。人物详情页顶部简介读取 `profile.profile_summary`；没有摘要时显示“档案摘要将在新的资料上传解析后自动生成。”可信度数值和建议只在 `/personas/{id}/uploads` 展示；`POST /api/personas/{id}/recalculate-trust` 作为兼容 API 保留，会重新运行 `memory_document_generation` 并创建 succeeded `calculate_trust_score` AI Job。

### 7. 在 Agent 交互中使用记忆

当前已实现文本对话中的基础记忆引用。进入 `/personas/{id}/chat` 后，页面会加载或创建当前人物的 conversation，用户可以发送文本消息；后端使用人物关系、称呼、说话风格、情绪风格、禁用表达、profile summary、长期记忆 Markdown、短期对话 Markdown 和对话历史生成第一人称回复。长期 Markdown 来自 confirmed/corrected 记忆，短期 Markdown 按人物聚合所有未删除 conversation messages；Markdown 过长时通过 `memory_context_compression` 压缩，失败时确定性截断。`/personas/{id}/regrets` 会加载或创建 `kind=regrets` 的专用 conversation，system prompt 聚焦以前没说的话、道歉、感谢、想念、告别或心结，且只读取 regrets 同类短期 Markdown；`/personas/{id}/wishes` 会加载或创建 `kind=wishes`、`context_kind=wishes` 的专用 conversation，system prompt 聚焦心愿、替 TA 继续做的一件事和下一步行动，并要求偏题时温和拉回该主题；wishes 不读取普通聊天或遗憾对话的短期 Markdown，也不使用默认记忆召回/引用。非 test 环境且 MiniMax/OpenAI-compatible key、base URL 与 `OPENAI_MODEL` 配置齐全时，`chat_llm` 走 MiniMax；如果 MiniMax 文本生成失败且 OpenAI-Next fallback 配置齐全，`chat_llm` 会追加一次 OpenAI-Next 文本兜底；二者都不可用时回到 deterministic mock 或既有服务层 fallback。涉及具体记忆时，回复下方可展开依据，显示引用的记忆和来源位置；用户可在依据区纠正这条记忆，纠正后后续对话优先使用修正内容。

### 8. 使用默认 TTS（后端 API）

当前已实现后端默认 TTS 配置、mock/可选 MiniMax 语音合成、音色样本和 mock/可选 MiniMax 音色克隆 API。`GET /api/personas/{id}/voice` 可查看声音状态；初始状态为 `no_voice`。`POST /api/personas/{id}/voice/default-tts` 可选择系统默认 TTS，并返回提示：

```text
当前使用系统默认声音，不是 TA 的真实声音。上传 TA 的清晰语音后，可以尝试生成模拟音色。
```

`POST /api/personas/{id}/voice/samples` 可从当前用户当前人物的音频资料创建 `sample_ready` 音色样本，并写入 `extract_voice_sample` AI Job。`POST /api/personas/{id}/voice/clone` 可发起 `voice_clone`；配置 MiniMax 时调用真实 provider，未配置或 test 环境使用 mock，成功时返回 selected `cloned_ready` 和 model URL，失败时标记 `clone_failed` 并回退 selected `default_tts`。后端在 clone 前会对可读 WAV 做 MiniMax 最少 10 秒预检，短音频会写入 `voice_preflight` failed job 并返回“至少 10 秒”错误，不再把明显不合规录音发给 MiniMax。`POST /api/personas/{id}/voice/synthesize` 会通过 `tts` Provider Gateway 返回音频 URL，并创建 `synthesize_speech` AI Job。`POST /api/conversations/{id}/voice-message` 可从当前用户当前 conversation 下的音频资料执行 ASR（未配置 DashScope 时为 mock）、文本 chat（配置 MiniMax 时为真实 `chat_llm`）和 TTS，把 `audio_url` 写入 persona message；相关 `asr_audio` 与 `synthesize_speech` 任务可在 `/api/personas/{id}/jobs` 查看。前端 `/personas/{id}/voice` 可设置默认 TTS，并在声音页直接上传纯净无噪声的人声音频或调取浏览器麦克风录制 TA 的人声音频来创建音色样本；页面提示音频必须只包含 TA 的清晰人声、避免背景音乐、多人同时说话和明显环境噪声，录音停止后会先展示预览，同时把提交文件转为 `voice-sample-recording-{timestamp}.wav`，并在上传创建样本前检查 MiniMax 克隆样本至少 10 秒，用户确认后才上传并创建样本。该页保留两个步骤：“上传并创建音色样本”只创建可克隆样本，“生成模拟音色”才调用 MiniMax 克隆并生成可用于 TTS 的 voice_id。聊天页可使用浏览器录音上传音频，也可选择已上传音频发送语音消息并播放返回音频。当前文本 chat、TTS 和音色已可走 MiniMax，但真实质量仍需按账号、模型和样本单独验收；ASR 质量需配置 DashScope 后单独验收。

当前已实现后端 3D 数字人配置 API、GLB 上传/文件读取 API 和前端 GLB 形象设置页。`GET /api/personas/{id}/avatar` 可查看当前形象状态；初始状态为 `no_avatar`。`POST /api/personas/{id}/avatar/upload` 使用 `multipart/form-data` 字段 `file` 上传自包含 `.glb`，后端保存到本地 `storage/avatar_models/{user_id}/{persona_id}/`，Compose 本地开发下该目录由 `avatar-model-storage` 卷持久化；接口创建 selected `AvatarModel`，状态为 `uploaded_ready`、格式为 `glb`，不会创建 AI Job 或进入资料解析队列。`GET /api/avatar-models/{id}/file` 按 JWT 用户隔离返回 `model/gltf-binary` 文件；跨用户、已删除或文件不存在时返回 404。旧 `POST /api/personas/{id}/avatar/default` 和 `POST /api/personas/{id}/avatar/generate` 仅作为兼容接口保留，不再是当前前端流程。

前端 `/personas/{id}/avatar` 只提供 GLB 上传、上传中/失败、模型加载中/成功/失败、当前模型记录和模型地址展示。聊天页 `/personas/{id}/chat`、遗憾对话室和心愿延续页会读取 selected avatar model；右侧数字人舞台无模型时展示空态，有 GLB 时通过共享 `AvatarPreview`/`AvatarStage` 使用 GLTFLoader + MeshoptDecoder 加载同一后端模型文件，支持常见 `EXT_meshopt_compression` 压缩 GLB。`AvatarPreview` 使用 Three.js `AnimationMixer` 优先播放 GLB 自带的 idle/wave/nod/listen/speak 等匹配动画；缺少对应 clip 时使用整体转身、点头、轻摆、靠近和光效作为基础反馈，并在 UI 中提示当前模型使用 fallback。聊天页视频手势模式使用浏览器本地摄像头和 `@mediapipe/tasks-vision` 识别手掌/握拳、手部轨迹和停留状态，再把结果映射为 `motionIntent` 驱动数字人动作；摄像头视频不上传后端、不写入数据库。当前不包含复杂手语、全身动作捕捉、多人识别、手势触发模型回复、真实音频音量包络或 viseme 口型同步，也不代表真实 3D 生成 provider 质量。

### 9. 生成回忆讲述

当前已实现后端故事生成、列表、收藏、默认 seed 和导出基础 API。前端不再暴露独立 `/personas/{id}/stories` 回忆讲述页；回忆讲述已放入 `/personas/{id}/memories` 记忆档案馆，页面进入时通过 `POST /api/personas/{id}/stories/seed` 幂等补足最多三段默认回忆故事，每段锚定一条 confirmed/corrected 且未删除的真实 MemoryCard；用户可在该页输入关键词并追加生成新的回忆讲述。`POST /api/personas/{id}/stories` 仍可从当前用户当前人物的 confirmed/corrected 记忆生成第一人称故事：后端用用户主题读取长期/短期记忆 Markdown，过滤掉未审核、停用、删除或越权来源 ID，并在入库、列表、导出和 TTS 前清洗 MiniMax/provider 输出中的 `<think>` 或思考文本；如果模型输出清洗后为空，则回退中文 deterministic story。接口返回 `source_memory_ids`、来源摘录/位置、mock `audio_url`，并创建 `generate_story` 与 `synthesize_speech` AI Job；如果没有已确认或已修正记忆，手动生成会返回 400，seed 接口返回现有故事空列表，避免根据未审核资料编故事。`GET /api/personas/{id}/stories` 可列出故事，`POST /api/stories/{id}/favorite` 可更新收藏状态。`GET /api/personas/{id}/export/story/{story_id}` 可返回故事文本导出内容、text 文件名、mock 音频 URL、audio 文件名、来源记忆和 mock 音频提示；`GET /api/personas/{id}/export/story/{story_id}/audio` 可返回带 AI simulation metadata 与响应头提示的 mock WAV 音频文件。

### 10. 导出或审计

当前已实现故事文本导出、mock 音频链接导出、后端导出元数据 API、后端 mock WAV 音频文件导出 API，以及档案/记忆/对话 JSON 导出 API：`GET /api/personas/{id}/export/profile`、`GET /api/personas/{id}/export/memories`、`GET /api/conversations/{id}/export`。三类 JSON 导出都按当前用户隔离并包含「AI 模拟」水印说明；`DELETE /api/personas/{id}` 会按 PRD 软删除人物及其资料、解析块、记忆、档案、对话消息、引用、声音模型、3D 模型、AI Job 和故事记录，已删除 AI Job 不再可由任务详情接口读取，并会在数据库提交后删除对应本地 `storage/materials/...` 资料文件；`DELETE /api/conversations/{id}` 会软删除单条对话及其消息、引用，并让对话列表、消息、引用和对话导出不再返回该对话；`DELETE /api/materials/{id}` 会软删除资料并删除对应本地资料文件；`DELETE /api/settings/data` 会软删除当前账号下所有人物域数据、删除当前账号对应本地资料文件，但保留用户账号和当前 token 可继续访问 `/api/auth/me`。`GET/PUT /api/settings/providers` 可查看和更新本地 provider 运行配置状态，写入目标是 git 忽略的 `.env/runtime.env`，响应不返回原始密钥。当前前端不暴露 `/settings/data` 或 `/settings/providers` 页面；真实 MinIO/S3 对象删除、审计回放或版本查看仍未实现。后端已有 `AuditLog` 模型骨架供后续实现使用。

| `/api/personas/{id}/voice/default-tts` | 默认 TTS API | 为当前用户人物选择系统默认 TTS，并明确标注不是 TA 的真实声音 |
| `/api/personas/{id}/voice/samples` | 音色样本 API | 从当前用户当前人物的音频资料创建 sample_ready VoiceModel，并写入 `extract_voice_sample` AI Job |
| `/api/personas/{id}/voice/clone` | 音色克隆 API | 使用 mock/可选 MiniMax `voice_clone` 生成 cloned_ready VoiceModel；可读短 WAV 会先以 `voice_preflight` 失败，其他失败时回退默认 TTS |
| `/api/personas/{id}/voice/synthesize` | 文本转语音 API | 使用 mock `tts` Provider Gateway 生成 mock audio URL，并创建 `synthesize_speech` AI Job |
| `/api/personas/{id}/avatar` | 形象配置 API | 查看当前用户人物的 3D 形象状态、形象记录、兼容选项和失败提示 |
| `/api/personas/{id}/avatar/upload` | GLB 形象上传 API | 上传自包含 `.glb`，创建 selected `uploaded_ready` AvatarModel，不创建 AI Job |
| `/api/avatar-models/{id}/file` | GLB 模型文件 API | 按 JWT 用户隔离返回 `model/gltf-binary`，跨用户、已删除或文件不存在返回 404 |

| `/api/personas/{id}/avatar/generate` | 3D 形象生成兼容 API | 保留旧图片资料 mock 生成能力；当前前端流程不再暴露 |
| `/api/personas/{id}/stories` | 故事列表/生成 API | 列出当前用户人物的故事，或按主题读取长期/短期记忆，从 confirmed/corrected 记忆生成已清洗思考块的第一人称故事、来源和 mock 音频 |
| `/api/personas/{id}/stories/seed` | 默认故事 seed API | 幂等补足最多三段默认回忆故事，每段锚定一条 confirmed/corrected 且未删除记忆；无已审核记忆时返回现有故事列表而不编造 |

## 技术方案

当前已确认的技术方案：

- 前端：Next.js 15、React 19、TypeScript、Tailwind CSS、Three.js、`@mediapipe/tasks-vision`；当前已接入星空视觉组件、auth/demo helper、persona helper、materials/jobs/memories/audit/profile/chat/voice/avatar/stories API helper、顶部四入口、人物总览功能卡片、资料解析与审核页、uploads 内可信度和记忆星标审核、任务状态页、记忆档案馆、文本/视频手势/语音对话页、遗憾对话室、心愿延续系统、声音设置页、3D 形象设置页和基础表单校验。登录/注册、数据设置、模型设置、独立 profile UI 和独立 stories 页面当前不暴露。
- 后端：FastAPI、SQLAlchemy、Alembic、JWT Auth、用户隔离的人物 CRUD 与人物删除级联软删除、SourceMaterial API、AI Job API、mock parsing service、Memory Audit v2 API、memory conflict API、semantic search API、profile/Persona Engine API、memory_document_generation trust API、text chat API、单条 conversation 软删除、清空当前账号数据、模型设置运行配置 API、voice/default TTS/samples/clone/synthesize/voice-message API、stories generate/list/favorite/export/audio-file export API 和 profile/memories/conversation JSON export API。
- 数据存储：PostgreSQL、Redis、MinIO 通过 Docker Compose 接入；后端容器启动前执行 Alembic 迁移；测试使用 SQLite 内存库；当前资料上传文件使用仓库本地 `storage/materials/...` 作为开发兜底，GLB 形象文件使用 `storage/avatar_models/...` 并在 Compose 下挂载 `avatar-model-storage`。ECS 直连端口方案只对外开放前端 `3000` 和后端 `8000`，数据库、Redis 和 MinIO 宿主端口默认仅绑定到 `127.0.0.1`。
- AI 能力：当前通过 provider gateway 返回 deterministic mock parsing、严格六模块 JSON memory extraction、`chat_llm`、`memory_context_compression`、`persona_profile_analysis`、`tts`、`extract_voice_sample`、`voice_clone`、`avatar_3d` 和 `story_generation` 输出；配置 DashScope 后，解析链路的 `text_parser`、`ocr`、`image_understanding`、`asr`、`video_understanding` 和 `memory_extraction` 可调用 Qwen/DashScope；配置 MiniMax 后，`chat_llm`、`story_generation`、`memory_context_compression`、`persona_profile_analysis`、`memory_document_generation`、TTS 和 voice clone 可走 MiniMax adapter，其中文本能力使用 OpenAI-compatible `/chat/completions`；配置 OpenAI-Next 后，仅当 MiniMax 文本 capability 失败时追加一次 text-only fallback，不参与 TTS、voice clone 或多模态解析；自动记忆审核刷新 profile 和长期 Markdown，不覆盖上传解析链路生成的唯一 trust；真实 Chat/Story/Persona Engine 长期质量、真实 TTS/voice clone 质量和 3D provider 尚未完成系统验收。
- 测试与部署：后端 pytest、前端 test/lint/build、`docker compose config`；ECS 部署使用 `.env/runtime.env` 填写 `FRONTEND_URL`、`BACKEND_URL`、`NEXT_PUBLIC_API_BASE_URL` 和 `JWT_SECRET` 后运行 `docker compose --env-file .env/runtime.env up --build -d`；`docs/init.sh` 仅在用户明确要求完整基线验证时运行。
