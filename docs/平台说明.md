# 可信人格记忆Agent 平台说明

本文档面向第一次使用可信人格记忆Agent的开发者或评测人员，说明当前准备事项、启动方式、验证方式和常见结果。当前后端覆盖 Milestone 0-8 的健康检查、认证/demo API、人物、资料、任务、Memory Audit v2 记忆档案馆、档案/可信度、对话、声音、3D、stories/export/delete/provider settings API，本地资料文件清理、长期/短期记忆 Markdown 和可选 DashScope/MiniMax provider 接入。当前前端以 `origin/yawen` 的星空交互为准：展示星空首页、Dashboard、创建星星、人物详情、资料上传、任务、记忆解析与审核、人格档案、对话、声音和 3D 形象页；对话页右侧提供数字人舞台，形象页生成或选择 avatar 后会替换默认星空占位；审核页展示可信度横幅、6 个维度卡片和底部完成审核按钮。不展示登录/注册、登录态差异、数据设置页、模型设置页或独立回忆讲述页。无 token 时前端会透明调用现有 `POST /api/auth/demo` 获取 demo token；已有 token 静默复用。真实解析质量需要本地配置 `DASHSCOPE_API_KEY` 后测试；真实文本 Chat/Story LLM、Persona Engine 显式画像分析和记忆上下文压缩已可使用本地 MiniMax/OpenAI-compatible 配置，长期人格质量、真实 TTS/音色质量、真实 GLB/3D provider 质量、真实 MinIO/S3 对象删除和生产级 secret manager 仍待后续里程碑接入。

补充：embedding 运行入口已停用并移除；历史数据库 embedding 列和 Alembic `0004` 保留以兼容旧库。当前对话上下文链路为“LLM/规则抽取结构化记忆 -> 已确认/已修正记忆生成长期 Markdown -> 人物维度短期对话 Markdown -> Chat 直接读取 Markdown”。Markdown 过长时通过 `memory_context_compression` 压缩，失败时用确定性截断兜底，不阻断聊天。

## 适用对象与使用前准备

### 适用对象

- 开发者：验证本地基础工程、后端测试、前端检查、Compose 配置、Milestone 1-8 后端/API 基础闭环，以及当前 yawen 星空前端的首页、Dashboard、创建星星、人物详情、资料上传、任务、记忆档案馆、profile/trust、chat、voice 和 avatar 页面。Milestone 8 stories/export/delete/provider settings 后端 API 保留，但当前不暴露独立前端设置页或 stories 页。
- 评测人员：确认当前是否具备前后端基础连通、健康检查、注册/登录 API、人物 CRUD、人物列表、人物创建表单校验、人物记忆空间页、资料上传页、任务状态页、记忆档案馆、人格档案/可信度页和文本对话页。
- 产品体验者：当前可以从星空首页进入体验，无需主动登录或注册；系统会在需要鉴权时自动创建 demo session。前端可创建星星、上传珍贵回忆、查看资料任务、点亮/审计记忆、查看人格档案、进入三类对话模式、使用声音设置和 3D 形象设置。上传资料会生成 deterministic mock 记忆卡片供确认、修改、拒绝、禁用或删除，审核页以 6 个维度卡片组织可见记忆，审核后 profile/trust 会按 local deterministic 规则刷新；文本对话会使用 confirmed/corrected 记忆并展示引用依据，配置 MiniMax 后会通过真实 `chat_llm` 生成回复。对话页右侧数字人舞台会优先展示已选择/生成的 mock avatar，未设置时展示星空占位。对话页的记忆档案馆、遗憾对话室和心愿延续入口只作为前端模式或 prompt 入口，不代表已实现独立后端能力。当前仍不能体验真实 3D 模型质量；真实文本 LLM 和 TTS/音色质量只完成基础接入与烟测，仍需用更大样本单独验收。

### 使用前准备

当前需要具备：

- 能打开本仓库。
- Python 3.12 或兼容环境，可安装 `backend/requirements.txt`。
- Node.js 和 npm；在 PowerShell 中运行 npm 命令时使用 `npm.cmd`。
- Docker Desktop 或兼容 Docker Compose，用于验证 Compose 配置和本地服务拓扑。

不需要提交真实密钥。Compose 和后端配置只使用变量名、空白 provider 配置和开发默认值；真实 provider key 应只放在本地未提交环境中。默认 `DEFAULT_LLM_PROVIDER=mock` 不会调用第三方；如需真实解析，可在 `.env/runtime.env` 中配置 `DEFAULT_LLM_PROVIDER=dashscope`、`DASHSCOPE_API_KEY`、`DASHSCOPE_WORKSPACE_ID`、`DASHSCOPE_COMPAT_BASE_URL`、`QWEN_TEXT_MODEL`、`QWEN_VISION_MODEL`、`QWEN_OCR_MODEL` 和 `QWEN_ASR_MODEL`。如需真实文本 Chat/Story LLM，使用本地 MiniMax 配置：`MINIMAX_API_KEY`、`MINIMAX_BASE_URL=https://api.minimaxi.com/v1`、`OPENAI_BASE_URL=https://api.minimaxi.com/v1`、`OPENAI_API_KEY` 和 `OPENAI_MODEL`；响应和文档只显示配置状态，不输出原始 key。当前前端不暴露模型设置页；如需查看或更新 provider configured/missing 状态，可通过后端 `GET/PUT /api/settings/providers` 或本地 `.env/runtime.env` 处理，响应只返回遮蔽后的配置状态，不返回原始 API key。

如需执行真实多模态公网样本烟测，先启动后端，再运行：

```powershell
python backend/scripts/real_multimodal_smoke.py --sample-mode public --backend-url http://localhost:8000
```

该命令会下载公网公版样本、派生 PDF/DOCX/DOC、本地裁剪视频，并真实调用 DashScope；结果写入 `.smoke/real-multimodal/results/`，不会提交或输出原始密钥。当前本地最近一次完整烟测已在 2026-07-04 通过，结果为 `.smoke/real-multimodal/results/20260704-170719.json`：text、image、audio、video、pdf、docx、doc 均生成 DashScope third_party parse job、ParsedChunk 和 source-backed MemoryCard，`errors=[]`。如果复跑时再次出现 `403 AccessDenied.Unpurchased` / `Access to model denied. Please make sure you are eligible for using the model.`，先在阿里云百炼控制台确认账号已开通百炼服务、完成实名认证、业务空间/API Key 与当前 region 匹配，并确认 `QWEN_TEXT_MODEL`、`QWEN_VISION_MODEL`、`QWEN_OCR_MODEL`、`QWEN_ASR_MODEL` 对该 key 可调用；开通或替换为有权限的模型后重启 backend 再复跑。

### 启动与访问

安装前端依赖：

```powershell
npm.cmd --prefix frontend install
```

如用户明确要求完整基线验证，可运行：

```powershell
& "C:\Program Files\Git\bin\bash.exe" ./docs/init.sh
```

启动本地服务拓扑：

```powershell
docker compose up --build
```

后端容器启动时会先执行 `alembic upgrade head`，再启动 Uvicorn 服务；因此 Compose 中的注册/登录 API 会使用已迁移的 PostgreSQL 表结构。

默认访问地址：

- 前端：http://localhost:3000
- 后端健康检查：http://localhost:8000/health
- MinIO Console：http://localhost:9001

## 用户全流程操作步骤

以下只描述当前可验证流程。完整人格记忆产品流程需在后续里程碑实现后补齐。

### 1. 进入系统

启动 Compose 后打开 http://localhost:3000 查看星空首页；访问 http://localhost:8000/health，返回 `{"status":"ok"}` 表示后端健康检查可用。当前前端不提供 `/login` 或 `/register` 页面；无 token 时，创建和核心工作流会自动调用 `POST /api/auth/demo` 创建本地演示用户，无需邮箱或密码。

### 2. 配置或确认人格记忆范围

当前已实现人物创建基础配置。进入 `/dashboard` 可查看当前 token 对应的人物列表；进入 `/personas/new` 可创建已故亲友、在世亲友或虚拟角色，并填写姓名、年龄/享年、简介、TA 对用户的角色和 TA 对用户的称呼。主要语言固定为中文；说话风格、情绪陪伴方式和禁用表达由系统默认值写入。缺少必填字段或年龄/享年超出 1..150 时，前端会阻止提交；后端也会拒绝缺少或无效的 age，并拒绝非 `zh-CN` 的语言值。创建成功后，创建页会把已选择的照片、视频、声音和文字文件通过现有资料上传 API 补传；若上传失败，已创建人物会保留，用户可进入资料上传页继续补传。

### 3. 输入或导入记忆材料

当前已实现资料记录、默认 mock 解析和可选 DashScope 真实解析。登录或进入演示后，从人物记忆空间点击「补充资料」进入 `/personas/{id}/uploads`，可上传文本、图片、音频、视频文件，也可填写手动资料。提交后系统创建 SourceMaterial 记录，把上传文件写入仓库本地 `storage/materials/...` 兜底路径；默认通过 deterministic mock Provider Gateway 创建 `ParsedChunk` 和带来源的 `MemoryCard`，配置 `DEFAULT_LLM_PROVIDER=dashscope` 与 `DASHSCOPE_API_KEY` 后，文本/PDF/DOC/DOCX 抽取、图片 OCR/理解、音频 ASR、视频音轨/关键帧解析和记忆抽取会走 DashScope adapter。

### 4. 查看记忆提取结果

当前已实现记忆提取结果和审计状态展示。进入 `/personas/{id}/memories` 可查看星空主题“记忆解析与审核”：页面先展示可信度横幅、重新解析/新增维度按钮、6 个维度卡片和底部“完成审核，点亮星星”按钮；系统从资料中整理出的真实后端记忆仍按分类进入这些维度卡，维度确认/删除由前端批量调用现有记忆 API。页面下方继续显示健康分、待审核数、开放冲突、来源覆盖、最近事件、语义搜索结果、冲突中心和选中记忆历史。进入 `/personas/{id}/jobs` 可查看资料对应的 AI Job 状态、任务类型、重试次数、provider 和错误信息；默认解析为同步 mock 输出，配置 DashScope 后可进行真实 provider 调用，真实质量需用有效 key 和真实材料单独验收。

### 5. 复核、修正或删除记忆

当前已实现记忆复核、修正、删除和审计追踪。进入 `/personas/{id}/memories` 后，可以确认、编辑标题/内容、拒绝、禁用或删除记忆；编辑会保存为用户修正后的内容。Memory Audit v2 会在同一事务中记录创建、编辑、确认、拒绝、停用、删除、搜索、对话引用、对话纠错、故事引用、profile 编辑/重生成和可信度变化事件；冲突中心可把开放冲突处理为 `resolved_by_user` 或 `dismissed`。Milestone 4 后这些状态会刷新人格档案和可信度；confirmed/corrected 记忆会写入该人物长期记忆 Markdown，rejected/disabled/deleted 记忆不会进入长期 Markdown 或对话引用。

### 6. 查看和编辑人格档案与可信度

当前已实现 deterministic local 人格档案、Persona Engine 显式画像分析和可信度页。进入 `/personas/{id}/profile` 可查看可信度总分、等级、PRD 权重组件、上传建议、profile summary、各档案维度和来源记忆 ID；页面支持保存手动档案修改、从已审核记忆重生成档案，以及手动重算可信度。显式重生成会调用 `persona_profile_analysis` capability，输入 persona card、已解析 chunk、active memory cards 和 source metadata，保存完整 `persona_engine_json` 与生成时间；测试环境和未配置真实模型时走 deterministic mock，解析失败时回退 deterministic profile，不影响聊天 prompt。显式重生成和重算会创建 succeeded `update_profile` 与 `calculate_trust_score` AI Job。

### 7. 在 Agent 交互中使用记忆

当前已实现文本对话中的基础记忆引用。进入 `/personas/{id}/chat` 后，页面会加载或创建当前人物的 conversation，用户可以发送文本消息；后端使用人物关系、称呼、说话风格、情绪风格、禁用表达、profile summary、长期记忆 Markdown、短期对话 Markdown 和对话历史生成第一人称回复。长期 Markdown 来自 confirmed/corrected 记忆，短期 Markdown 按人物聚合所有未删除 conversation messages；Markdown 过长时通过 `memory_context_compression` 压缩，失败时确定性截断。非 test 环境且 MiniMax/OpenAI-compatible key、base URL 与 `OPENAI_MODEL` 配置齐全时，`chat_llm` 走 MiniMax，否则回到 deterministic mock。涉及具体记忆时，回复下方可展开依据，显示引用的记忆和来源位置；用户可在依据区纠正这条记忆，纠正后后续对话优先使用修正内容。

### 8. 使用默认 TTS（后端 API）

当前已实现后端默认 TTS 配置、mock/可选 MiniMax 语音合成、音色样本和 mock/可选 MiniMax 音色克隆 API。`GET /api/personas/{id}/voice` 可查看声音状态；初始状态为 `no_voice`。`POST /api/personas/{id}/voice/default-tts` 可选择系统默认 TTS，并返回提示：

```text
当前使用系统默认声音，不是 TA 的真实声音。上传 TA 的清晰语音后，可以尝试生成模拟音色。
```

`POST /api/personas/{id}/voice/samples` 可从当前用户当前人物的音频资料创建 `sample_ready` 音色样本，并写入 `extract_voice_sample` AI Job。`POST /api/personas/{id}/voice/clone` 可发起 `voice_clone`；配置 MiniMax 时调用真实 provider，未配置或 test 环境使用 mock，成功时返回 selected `cloned_ready` 和 model URL，失败时标记 `clone_failed` 并回退 selected `default_tts`。`POST /api/personas/{id}/voice/synthesize` 会通过 `tts` Provider Gateway 返回音频 URL，并创建 `synthesize_speech` AI Job。`POST /api/conversations/{id}/voice-message` 可从当前用户当前 conversation 下的音频资料执行 ASR（未配置 DashScope 时为 mock）、文本 chat（配置 MiniMax 时为真实 `chat_llm`）和 TTS，把 `audio_url` 写入 persona message；相关 `asr_audio` 与 `synthesize_speech` 任务可在 `/api/personas/{id}/jobs` 查看。前端 `/personas/{id}/voice` 可设置默认 TTS、创建样本、发起克隆并播放预览；聊天页可使用浏览器录音上传音频，也可选择已上传音频发送语音消息并播放返回音频。当前文本 chat、TTS 和音色已可走 MiniMax，但真实质量仍需按账号、模型和样本单独验收；ASR 质量需配置 DashScope 后单独验收。

当前已实现后端 3D 数字人基础 API 和前端 3D 形象设置页。`GET /api/personas/{id}/avatar` 可查看当前形象状态；初始状态为 `no_avatar`。`POST /api/personas/{id}/avatar/default` 可选择默认纪念形象，返回 mock GLB、预览图、基础表情、待机/眨眼/微笑/点头动作配置和 audio envelope 口型配置。`POST /api/personas/{id}/avatar/generate` 可从当前用户当前人物的图片资料创建 `avatar_3d` AI Job，通过 mock `avatar_3d` Provider Gateway 生成 `generated_ready` AvatarModel；生成失败时写入 `generation_failed` 并回退默认纪念形象，返回提示：

```text
这张照片暂时没有生成成功。你可以换一张更清晰的正脸照，或者先使用默认纪念形象继续对话。
```

前端 `/personas/{id}/avatar` 可查看状态、选择默认形象、选择图片资料发起 mock 生成，并展示 Three.js 头像/半身预览、待机、眨眼、微笑、点头和简化口型测试。聊天页 `/personas/{id}/chat` 会读取 selected avatar model；右侧数字人舞台默认展示星空占位，模型可渲染时展示同一 mock 头像/半身数字人，若只有可显示 preview image 则展示预览图，并在 TA 的语音回复播放、暂停和结束时切换口型开合状态。当前不代表真实 3D 生成质量，不加载真实 GLB，也不包含真实音频音量包络或 viseme 口型同步。

### 9. 生成回忆讲述

当前已实现后端故事生成、列表、收藏和导出基础 API。feat-041 后，前端不再暴露独立 `/personas/{id}/stories` 回忆讲述页；相关体验入口整合在对话页的前端模式或 prompt 中，不声明独立后端新能力。`POST /api/personas/{id}/stories` 仍可从当前用户当前人物的 confirmed/corrected 记忆生成第一人称故事，返回 `source_memory_ids`、来源摘录、mock `audio_url`，并创建 `generate_story` 与 `synthesize_speech` AI Job；如果没有已确认或已修正记忆，会返回 400，避免根据未审核资料编故事。`GET /api/personas/{id}/stories` 可列出故事，`POST /api/stories/{id}/favorite` 可更新收藏状态。`GET /api/personas/{id}/export/story/{story_id}` 可返回故事文本导出内容、text 文件名、mock 音频 URL、audio 文件名、来源记忆和 mock 音频提示；`GET /api/personas/{id}/export/story/{story_id}/audio` 可返回带 AI simulation metadata 与响应头提示的 mock WAV 音频文件。

### 10. 导出或审计

当前已实现故事文本导出、mock 音频链接导出、后端导出元数据 API、后端 mock WAV 音频文件导出 API，以及档案/记忆/对话 JSON 导出 API：`GET /api/personas/{id}/export/profile`、`GET /api/personas/{id}/export/memories`、`GET /api/conversations/{id}/export`。三类 JSON 导出都按当前用户隔离并包含「AI 模拟」水印说明；`DELETE /api/personas/{id}` 会按 PRD 软删除人物及其资料、解析块、记忆、档案、对话消息、引用、声音模型、3D 模型、AI Job 和故事记录，已删除 AI Job 不再可由任务详情接口读取，并会在数据库提交后删除对应本地 `storage/materials/...` 资料文件；`DELETE /api/conversations/{id}` 会软删除单条对话及其消息、引用，并让对话列表、消息、引用和对话导出不再返回该对话；`DELETE /api/materials/{id}` 会软删除资料并删除对应本地资料文件；`DELETE /api/settings/data` 会软删除当前账号下所有人物域数据、删除当前账号对应本地资料文件，但保留用户账号和当前 token 可继续访问 `/api/auth/me`。`GET/PUT /api/settings/providers` 可查看和更新本地 provider 运行配置状态，写入目标是 git 忽略的 `.env/runtime.env`，响应不返回原始密钥。当前前端不暴露 `/settings/data` 或 `/settings/providers` 页面；真实 MinIO/S3 对象删除、审计回放或版本查看仍未实现。后端已有 `AuditLog` 模型骨架供后续实现使用。

## 页面含义

| 页面或入口 | 用途 | 用户可执行操作 |
| --- | --- | --- |
| `/` | 星空首页 | 进入星空体验入口；无 token 时后续工作流会自动创建 demo session |
| `/dashboard` | Dashboard 人物列表 | 查看当前 token 对应的人物列表，进入创建人物或人物详情 |
| `/personas/new` | 创建星星页 | 填写人物类型、基础资料、年龄/享年和关系称呼；语言固定中文，风格边界使用系统默认值；提交后创建人物并上传已选资料 |

## 技术方案

当前已确认的技术方案：

- 前端：Next.js 15、React 19、TypeScript、Tailwind CSS、Three.js；当前已接入星空视觉组件、auth/demo helper、persona helper、materials/jobs/memories/audit/profile/chat/voice/avatar/stories API helper、人物路由、资料上传页、任务状态页、记忆档案馆、人格档案/可信度页、文本/模式对话页、声音设置页、3D 形象设置页和基础表单校验。登录/注册、数据设置、模型设置和独立 stories 页面当前不暴露。
- 后端：FastAPI、SQLAlchemy、Alembic、JWT Auth、用户隔离的人物 CRUD 与人物删除级联软删除、SourceMaterial API、AI Job API、mock parsing service、Memory Audit v2 API、memory conflict API、semantic search API、profile/trust/Persona Engine API、text chat API、单条 conversation 软删除、清空当前账号数据、模型设置运行配置 API、voice/default TTS/samples/clone/synthesize/voice-message API、stories generate/list/favorite/export/audio-file export API 和 profile/memories/conversation JSON export API。
- 数据存储：PostgreSQL、Redis、MinIO 通过 Docker Compose 接入；后端容器启动前执行 Alembic 迁移；测试使用 SQLite 内存库；当前上传文件使用仓库本地 `storage/materials/...` 作为开发兜底。
- AI 能力：当前通过 provider gateway 返回 deterministic mock parsing、memory extraction、`chat_llm`、`memory_context_compression`、`persona_profile_analysis`、`tts`、`extract_voice_sample`、`voice_clone`、`avatar_3d` 和 `story_generation` 输出；配置 DashScope 后，解析链路的 `text_parser`、`ocr`、`image_understanding`、`asr`、`video_understanding` 和 `memory_extraction` 可调用 Qwen/DashScope；配置 MiniMax 后，`chat_llm`、`story_generation`、`memory_context_compression`、`persona_profile_analysis`、TTS 和 voice clone 可走 MiniMax adapter，其中文本能力使用 OpenAI-compatible `/chat/completions`；自动记忆审核刷新中的 profile/trust 使用 deterministic local 规则；真实 Chat/Story/Persona Engine 长期质量、真实 TTS/voice clone 质量和 3D provider 尚未完成系统验收。
- 测试与部署：后端 pytest、前端 test/lint/build、`docker compose config`；`docs/init.sh` 仅在用户明确要求完整基线验证时运行。
